# Architecture

This document describes how the Rust and Java sides of the library fit together, how threading works, and how callbacks flow between the two runtimes.

## The Bridge

The library is split into three layers:

```
+---------------------------+
|   Java Application Code   |
+---------------------------+
            |
            v
+---------------------------+
|   Java FFM Bindings       |   (generated by jextract from webrtc_ffi.h)
+---------------------------+
            |
            v
+---------------------------+
|   rust_webrtc_ffi.dll     |   (C ABI: no_mangle extern "C" functions)
+---------------------------+
            |
            v
+---------------------------+
|   webrtc-rs               |   (ICE, DTLS, SRTP, SCTP, RTP/RTCP, codecs)
+---------------------------+
```

Java never touches Rust structs directly. It passes numeric handles (`long` values) across the FFI boundary, and the Rust side maps those handles to the actual objects.

## Handle System

Every major object on the Rust side is wrapped in a `Box<T>` and leaked into a `*mut c_void` pointer. Java stores this as a `MemorySegment` (or raw `long`) and passes it back on every subsequent call.

```
Java:  long handle = webrtc_ffi_peer_create(config);
       webrtc_ffi_create_offer(handle);
       webrtc_ffi_peer_close(handle);   // frees the Box

Rust:  Box::into_raw(Box::new(Peer))    // create
       Box::from_raw(ptr as *mut Peer)   // close (drops the Box)
```

Handles are not thread-safe by themselves. The Rust side serializes access internally where needed.

## Threading Model

The Rust side runs two tokio runtimes:

| Runtime | Purpose | Thread count |
|---------|---------|-------------|
| `PC_RUNTIME` | Drives each peer connection and its callbacks | Multi-threaded |
| `CALL_RUNTIME` | Used only by `block_on` for FFI calls from Java | Multi-threaded (separate) |

Java calls into the FFI on the main thread. The FFI functions use `block_on` on `CALL_RUNTIME` to wait for async Rust work to complete. Results are returned synchronously to Java.

Callbacks from Rust (ICE candidates, data channel messages, track events) fire on `PC_RUNTIME` worker threads. These callbacks reach Java through jextract upcall stubs.

**Important constraint**: Callbacks that originate from `PC_RUNTIME` must not call `block_on` again, since tokio panics on re-entrant runtime use. Functions like `add_ice_candidate`, `send_text`, and `send_bytes` work around this by spawning a fire-and-forget task on `PC_RUNTIME` and returning immediately.

## Callback Flow

```
Rust (PC_RUNTIME worker)          Java (upcall stub)
        |                                |
        v                                v
  on_ice_candidate  ──────────>  Observer.onIceCandidate()
  on_data_channel   ──────────>  Observer.onDataChannel()
  on_track          ──────────>  Observer.onTrack()
  on_connection_state_change ──> Observer.onConnectionStateChange()
  on_ice_gathering_state_change > Observer.onIceGatheringStateChange()
```

Each callback type has a corresponding C function pointer type in Rust and a jextract-generated upcall interface in Java. The Java `PeerConnection.Observer` interface is the public API for receiving these events.

## Data Channel Polling

When a remote peer creates a data channel, the Rust `Forwarder` receives the channel and spawns a dedicated polling task for it. This task reads messages in a loop and forwards them to Java via the `on_message` callback.

The poller handles these events:
- `Some(msg)` -- forwards the message to Java, then continues
- `OnOpen` -- notifies Java the channel is ready
- `OnClose` -- stops the poller
- `OnBufferedAmountLow`, `OnError`, `OnClosing`, `OnBufferedAmountHigh` -- ignored (poller keeps running)

Only `None` (channel closed) and `OnClose` terminate the polling loop.

## Track Event Polling

Track remotes work similarly. When a remote track is registered, a polling task reads from its event channel:

- `OnOpen` -- fires the Java `OpenCallback`, passing SSRC and RID
- `OnRtpPacket` -- forwards the RTP packet data to Java via `RtpCallback`
- `OnMute` / `OnUnmute` -- forwarded if callbacks are registered
- `OnEnding` / `OnEnded` / `OnError` -- terminal events

The Rust side uses global `HashMap<u32, ...>` maps keyed by monotonically increasing IDs for both local and remote tracks. Java holds onto these IDs and uses them in subsequent FFI calls.

## Memory Management

Every FFI function that allocates Rust memory has a corresponding `_free` function. The Java side wraps these in `AutoCloseable` or `try-with-resources` patterns:

```java
Configuration cfg = Configuration.create();   // allocates Config on Rust heap
// ... use cfg ...
cfg.close();                                  // calls webrtc_ffi_config_free
```

Session descriptions are cloned by `setLocalDescription` / `setRemoteDescription` (the Rust side copies the SDP into the peer connection's state). The original handle remains valid and can be reused -- for example, the same offer can be passed to both `setLocalDescription` on the offerer and `setRemoteDescription` on the answerer.

## Codec Registration

When a peer connection is created, `MediaEngine::default().register_default_codecs()` is called on the Rust side. This registers all codecs supported by webrtc-rs (25 codecs total: Opus, G.722, PCMU, PCMA, VP8, VP9, H.264, H.265, AV1, RTX, FEC variants, etc.).

Java can query the supported codecs via `WebRtc.listSupportedCodecs()`, which calls `webrtc_ffi_supported_codecs` on the Rust side and parses the pipe-delimited response.
